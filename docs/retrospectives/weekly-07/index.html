En Fase 2 el foco está en la implementación de las historias de usuario HU 12, HU 13 y HU 14, que amplían la funcionalidad del sistema según el backlog y la planificación del proyecto.

✅ Qué salió bien

  • Arquitectura y organización: Monorepo bien estructurado con carpetas por módulo (inventory-service, api-gateway, frontend-landing, Portal, database, Docs). Separación clara de responsabilidades.
  • API Gateway: Configuración correcta con Spring Cloud Gateway; enrutamiento de /api/products/** al inventory-service. El frontend consume todo a través del Gateway (puerto 8082).
  • CRUD de productos: Implementación completa en el backend (Controller, Service, Repository, DTOs, entidades JPA). Endpoints accesibles vía Gateway.
  • Documentación: README claro con objetivo, arquitectura, tabla de servicios y puertos, orden de arranque. Carpeta Docs con architecture.md, Diagrama_del_Sistema.md, estimación MoSCoW y rúbrica de evaluación.
  • Metodología y priorización: Uso de Scrum (Jira), técnica MoSCoW aplicada al backlog y requisitos Must/Should/Could/Won't bien definidos para el MVP.
  • Infraestructura: Docker Compose con PostgreSQL y pgAdmin; persistencia con JPA y entidades mapeadas. Flujo Frontend → Gateway → Inventory Service → Base de datos documentado y probado.

🔧 Dificultades y qué se solucionó

  • Enrutamiento desde el Gateway: Asegurar que las peticiones llegaran correctamente al backend. Solución: Definir rutas en application.yml del api-gateway con predicado Path=/api/products/** y URI del inventory-service (8081).
  • Consistencia de capas y DTOs: Mantener separación entre entidades de dominio, DTOs de API (request/response) y capas Controller/Service/Repository. Solución: Estructura de paquetes clara y uso de DTOs (ProductResponse, ProductCreateRequest, ProductUpdateRequest) en los endpoints.
  • Manejo de errores en la API: Respuestas coherentes ante errores (404, 400, etc.). Solución: GlobalExceptionHandler y clases de excepción (NotFoundException, BadRequestException, ConflictException) con formato de error estandarizado (ApiError, ErrorResponse).
  • Orden de arranque de servicios: La base de datos, el inventory-service, el gateway y el frontend deben levantarse en un orden determinado. Solución: Documentación en README con tabla de puertos y comandos por carpeta (Docker, luego inventory-service, api-gateway, frontend-landing).
  • Alcance y priorización: Definir qué implementar primero en un contexto académico con tiempo limitado. Solución: Documento MoSCoW con Must Have (gestión productos, ventas, inventario, clientes, proveedores, reportes básicos) y planificación por fases; en Fase 2 nos enfocamos en la implementación de HU 12, HU 13 y HU 14.

📚 Qué nos quedó de aprendizaje

  • Arquitectura de microservicios: Ventajas de tener un API Gateway como único punto de entrada: simplifica el cliente, permite cambiar URLs internas y escalar servicios por separado.
  • REST y capas: La separación Controller → Service → Repository y el uso de DTOs mejoran la mantenibilidad y la claridad del contrato de la API.
  • Scrum y MoSCoW: Priorizar con Must/Should/Could/Won't y trabajar por historias de usuario (HU) ayuda a enfocar el esfuerzo y a presentar avances claros en la weekly.
  • Documentación desde el inicio: Tener README, arquitectura y diagramas actualizados facilita la onboarding, la presentación y la evaluación del proyecto.
  • Trabajo por fases e HUs: Organizar el desarrollo en fases (Fase 2: HU 12, HU 13, HU 14) permite entregar valor de forma incremental y mantener el backlog alineado con los objetivos del sprint.
